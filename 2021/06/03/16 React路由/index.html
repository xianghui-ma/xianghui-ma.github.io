16 React路由

前端路由及其实现原理

什么是前端路由呢？

前端路由为SPA而生。点击相应的路由链接就会显示与该路由相对应的内容，且不会刷新页面，同时你的每一次点击都会被加入浏览器的历史记录栈中以便进行回退/前进操作。因此使用前端路由便能在SPA页面中实现传统多页面的效果

前端路由的实现原理是什么呢？

前端路由的实现原理十分简单：监测浏览器地址栏中URL的变化，一旦URL发生变化，那么就会解析URL以匹配路由规则（即Route路由组件指定的路径path）并展示相应内容

但是需要注意的是：虽然地址栏中的URL会发生变化，但是并不会致使页面刷新（因为是SPA，所以必须保证页面不会刷新）

有两种方式能够实现上述原理：

1、hash路由

hash路由通过URL中的hash值（URL中#后面的内容）来匹配Route路由组件指定的路径path

hash路由实际上就是我们之前所学的锚点，锚点用于定位到当前页面中的某个位置且不会致使页面刷新

如上页面中有上述锚点，倘若点击test1，则浏览器地址栏中的URL变为E:/测试/index.html#/test1。URL中的#后的内容（即hash值）便被用于解析以匹配路由

我们可通过hashchange事件来监听URL中hash的变化，一旦变化便解析出hash值，再用该hash值匹配相应路由即可

注意：#后面的hash值不会作为参数发给服务器

2、history路由

history路由底层使用的是H5提供的操纵浏览器历史记录的API。即history路由操纵的是浏览器历史记录

H5中提供了两个操纵浏览器历史记录的新API——pushState和replaceState。当通过pushState（replaceState）向浏览器的历史记录中增添一个历史记录（替换历史记录栈顶的历史记录为一个新的历史记录）时，浏览器地址栏中的URL会改变，且不会致使页面刷新。我们可通过onpopstate事件来监听改变，从而解析URL以匹配路由规则并展示相应内容

例如我们在https://developer.mozilla.org/zh-CN/docs/Web/API/History/pushState页面下，然后打开控制台并运行上述代码，则URL变为https://developer.mozilla.org/test

对于前端路由更清楚的解释可见**前端路由**

hash路由与history路由都会将新的URL加入浏览器的历史记录栈中

React路由的基本使用

React路由有三种不同的版本——WEB/NATIVE/ANYWHERE——分别表示web版/native原生应用版/任何地方都能使用

由于我们开发的是web应用，所以使用的是WEB版，它对应的npm包为react-router-dom，我们要下载该包并引入

React常用路由组件介绍

具体可见**官网**

1、<BrowserRouter>与<HashRouter>

<BrowserRouter>使用的是前面所说的history路由；<HashRouter>使用的是hash路由

这两个组件用于作为根路由组件——即其他所有路由组件都应是跟路由组件的子组件。我们一般将它们包裹在App组件外

2、<NavLink>与<Link>

它们用于注册路由链接，他们的to属性用于指定要连接到的路由（路径）

<NavLink>实际上是特殊版本的<Link>，特殊在其可以给被点击的路由链接添加样式，使用方法是：

activeClassName用于指定当路由链接被点击时，向该元素追加的class属性，这样便可以通过class属性为被点击的路由链接设置样式

3、<Switch>

React在进行路由匹配时，其会将路由链接与所有<Route>的path进行比对，如果发现了多个匹配的路由，那么React会渲染多个组件进行展示。但通常我们都是一个路由链接匹配一个<Route>，因此一旦匹配了一个路由后便没必要继续向下匹配。这时我们可用<switch>组件包裹住所有<Route>组件，这样在匹配到第一个路由后就不会继续向下匹配了（如同**官网**中的描述：All children of a <Switch> should be <Route> or <Redirect> elements. Only the first child to match the current location will be rendered.）

如上，如果被点击的路由链接是/home，那么仅会匹配第一个<Route path='/home' component={Home}/>后便停止向后匹配

需要注意的是，<Switch>的子元素仅能是<Route> 或 <Redirect> ，具体见上述官网

4、<Redirect>

Redirect起着兜底的作用，即如果没有Route组件的path与路由链接的路径相匹配，则转向Redirect指定的路径

当点击<Link to='/home/a/b'>Home</Link>时，由于没有Route进行匹配，所以重定向到/about路由

需要注意的是，<Redirect to="..."></Redirect>应与<Switch>组件配合使用，且<Redirect to="..."></Redirect>中to指定的重定向路由必须是<Switch>组件中出现的路由（如上述代码改为<Redirect to="/home"></Redirect>，但<Switch>组件中没有路径为/home的Route组件，故不能正常工作）

Redirect还可用来设置默认选中哪个路由，页面初次加载时，默认选中的路由就是Redirect指定的路由。从而展示页面在初次加载时你想呈现的内容

路由组件使用展示

Link/Route等路由组件需要包裹在BrowserRouter（使用history路由）/HashRouter（使用hash路由）路由组件中，且一般我们直接在最外层包裹上BrowserRouter/HashRouter路由组件

需要注意的是，像下面这种写法是错误的

因为虽然Link/Route等路由组件包裹在了BrowserRouter/HashRouter路由组件中。但是此时他们分属于两个BrowserRouter/HashRouter，因此彼此是独立的，所以即使你点击了路由链接也不会匹配到相应路由以展示相应组件。因此要想其他各路由组件间相互作用，那么就必须将他们包裹在同一个BrowserRouter/HashRouter中

路由的精确匹配与模糊匹配

<Route>组件有一个exact属性，为true时表示精确匹配，为false时（默认值）表示模糊匹配。模糊匹配时，路由链接的路径必须大于<Route>组件的路径，且模糊匹配的原则是：<Route>组件的路径是路由链接的路径的子串，且该字串的起始位置从路由链接的路径的第一个字符开始

在非必要时，不要使用精确匹配

replace模式

无论是history路由还是hash路由，默认情况下点击Link/NavLink路由链接都会向浏览器的历史记录栈中推入一个记录。但是我们可以开启replace模式，在该模式下，点击Link/NavLink路由链接会替换浏览器的历史记录栈的栈顶记录，而不是向栈中推入一个记录，因此历史记录栈中永远都只有一个记录，故便不能使用回退/前进按钮

如果要开启replace模式，则设置Link/NavLink的replace属性为true即可，即如<Link to="..." replace={true} />

1、写法不同

2、存放位置不同

在项目的文件夹结构中，我们将一般组件放于src/components文件夹下，而将路由组件放于src/pages文件夹下

编程式路由

什么是编程式路由

我们前面所说的路由都是需要Link/NavLink注册路由链接，然后点击路由链接才能跳转到相应路由。而编程式路由则是通过使用history（见前面的图片）对象中的API来工作的，因此即使我们不点击Link/NavLink路由链接也能实现路由跳转

如何使用

由于history是React路由自动传给路由组件的props的，因此我们只能在路由组件中使用编程式路由（因为非路由组件的props中没有history对象）

调用history中的push/replace方法便可以跳转到相应路由。需要注意的是：编程式路由也需要去匹配Route注册的路由

编程式路由也能够传递params/search参数，这些都同“向路由组件传递数据的方法”一样。只是需要注意的是，对于state参数而言，push/replace的第二个参数专门用于传递state参数

history中的其他API

history中的goBack/goForward/go分别用于后退/前进/跳转（传入负数后退，传入正数前进）

withRouter

在前面我们说到——路由组件的props中会有history/location/match属性。那么如果想在一般组件的props中也使用这三个属性怎么办呢？这时便可通过withRouter对一般组件进行包装，然后一般组件的props也就有了路由组件的props才有的属性
